# 妥当性と認知バイアス

調査の落とし穴は「情報不足」より「自分の認知」のほうが大きい。結論を書く前に必ずこの章のチェックリストを通す。

## 妥当性の種類

- **Internal validity（内的妥当性）** — その研究内で因果推論が成り立つか。交絡因子を制御できているか
- **External validity（外的妥当性 / 一般化可能性）** — 別の状況・環境・ユーザー層に適用できるか
- **Construct validity（構成概念妥当性）** — 測ろうとしたものを本当に測っているか（例: 「開発者体験」を行単位で測るのは怪しい）
- **Ecological validity** — 実験室ではなく、現実の運用条件で再現するか

**Reliability（信頼性）** と **Validity（妥当性）** を混同しない:

- Reliability: 同じ条件で同じ結果が出るか（反復精度）
- Validity: そもそも正しいものを測っているか

狂った時計は「正確に同じ時間を指す（高 Reliability）」が「正しい時間ではない（低 Validity）」。

## 反証可能性（Falsifiability, Popper）

主張が科学的であるためには「どんな観察なら私の主張は間違いだと認めるか」を**事前に言える**必要がある。これを言えない主張はリサーチの結論にならない。

例:

- 悪い主張: 「React は Vue より良い」（何をもって検証するか不明）
- 良い主張: 「我々のユースケース（SPA、中規模、TS前提）では、学習曲線（オンボーディング所要時間）と既存資産（チームの前職経験）の合計で React が優位。月次のバグ修正速度が Vue 採用時より 30% 以上遅ければこの主張は間違い」

## 再現性危機（Replication Crisis）

心理学・医学・経済学で2010年代に露呈した問題。有名な論文の結果が再現しない。ソフトウェア工学も例外ではない（ベンチマーク、A/B テスト、UXリサーチ）。

対策:

- **Pre-registration**: 仮説・方法を結果を見る前に公開登録する
- **Registered Reports**: 結果前に査読する（結果が有意でなくても出版される）
- **Artifact Review（ACM）**: コード・データを伴う査読
- **Replication studies**: 複数ソースで同じ結論が得られるか

ソフトウェア調査では「調査計画を issue に書いてから調べる」で代用可能。

## 統計的落とし穴

- **P-hacking**: 有意になるまで解析を繰り返す、データを切り刻む
- **HARKing**（Hypothesizing After Results are Known）: 結果を見てから仮説を立てる。Texas Sharpshooter Fallacy と同根
- **Multiple Comparisons**: 多重比較の補正（Bonferroni / Holm / FDR）を忘れて偽陽性を量産
- **Base Rate Fallacy**: 基準率を無視して条件付き確率を誤読
- **Simpson's Paradox**: 全体と部分で傾向が逆転する（グループ化で消える / 現れる効果）
- **Statistical vs Practical Significance**: 有意差があっても効果量が小さければ実用的には無視できる。逆に有意差が出なくても効果量が大きければ意味がある

## トライアンギュレーション（三角測量）

複数の視点・手法・データ源で同じ結論に到達するかを確認。Denzin の4分類:

1. **Data triangulation** — 複数データソース
2. **Investigator triangulation** — 複数調査者
3. **Theory triangulation** — 複数理論レンズ
4. **Methodological triangulation** — 複数手法

ライブラリ評価で「公式ドキュメント」「実コード」「コミュニティ事例」「ベンチ」を突き合わせるのは典型的なトライアンギュレーション。

## 認知バイアス一覧（最頻出10種）

### 1. 確証バイアス（Confirmation Bias）

信じたい仮説を支持する情報ばかりを集める。**最も強力で最も避けにくい**。

対策:

- 事前に反証条件を書く
- Steelman: 反対立場を最強の形で組み立ててから批判
- Red Team: 誰か（または自分）に反対役をさせる
- Devil's Advocate

### 2. サバイバーシップバイアス

成功事例ばかり見る。Abraham Wald の戦闘機補強の逸話（弾痕の多い部分ではなく、**戻ってこなかった機体の弾痕のない部分**を補強すべき）。

対策:

- 失敗事例を明示的に探す
- 「○○を採用して失敗した企業」「○○を辞めた企業」を調べる
- 「なぜ廃れたか」を問う

### 3. 選択バイアス / サンプリングバイアス

GitHub Stars、HN 投票、Twitter 反応は偏ったサンプル:

- 早期採用者の意見に偏る
- 英語圏エンジニアに偏る
- アーリーマジョリティより先
- 流行に反応しない silent majority を見落とす

### 4. アンカリング

最初に見た数字に引きずられる。

対策:

- 複数の独立見積もりを先に集めてから比較
- 最初の数字を「候補のひとつ」に相対化

### 5. 可用性ヒューリスティック

思い出しやすい事例を過大評価:

- 最近話題のライブラリ
- 派手な事故
- 自分の以前の経験

地味だが実証された選択肢を見落とす。

### 6. Dunning-Kruger 効果

知識が浅いほど自信過剰。浅いリサーチで結論を出した時ほど、根拠が薄いことを疑う。

### 7. Hindsight Bias

「あとから見れば明らかだった」という錯覚。過去の意思決定を評価する時、当時の情報だけで判断できたかを問う。

### 8. Streetlight Effect

街灯の下で鍵を探す。見つけやすい場所だけ探す。

- 英語圏・検索上位・公式ブログだけ見ていないか
- 日本語・非英語圏・古い論文・ソースコード・メーリングリストにも鍵はある

### 9. Texas Sharpshooter Fallacy

データを見てから的を描く。後付けで都合よくパターンを抽出。HARKing と同根。

### 10. Goodhart's Law / Campbell's Law

指標が目標になると、指標の意味が失われる。

- GitHub Stars 目的のライブラリ
- OSS コントリビューション数目的のフリップフロップ PR
- ベンチマーク目的の最適化

## バイアス対策の一般原則

- **プリコミット**: 仮説・評価基準を結果を見る前に書く
- **Devil's Advocate**: 誰かが反対役をする（自分でやってもいい）
- **Time Delay**: 熱狂が冷めるまで待つ
- **Outside View**（Kahneman）: 「このケースだけ特別」を疑う。類似案件の基準率を使う
- **Pre-mortem**: 失敗した未来から逆算する（「6ヶ月後この選択が失敗していたら、原因は何だったか」）

## 妥当性チェックリスト（結論を書く前に通す）

- [ ] 一次情報 / 元データにリンクされているか
- [ ] 反対意見・反証が1つ以上検討されたか
- [ ] サンプルサイズは十分か、選択バイアスはないか
- [ ] 調査時点 / 対象バージョンが明記されているか
- [ ] 利益相反（ベンダー、所属、スポンサー）は開示されているか
- [ ] 再現手順が書かれているか
- [ ] 「この観察が見えたら私の結論は間違い」と言えるか（反証可能性）
- [ ] 類似案件の基準率と比較したか（Outside View）
- [ ] 成功例だけでなく失敗例も検討したか
- [ ] 英語圏・検索上位以外も見たか
