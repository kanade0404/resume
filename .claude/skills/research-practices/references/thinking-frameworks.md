# リサーチの思考フレームワーク

問いを立て・分解し・バイアスを避け・結論に橋渡しするための思考道具。特定の流派に絞らず、古今東西から取捨選択してある。

## 問いを立てる

### イシュードリブン（安宅和人『イシューからはじめよ』）

良いイシュー（解くべき問題）の条件:

1. **本質的**: それを解けば他の多くが解決する、影響度が大きい
2. **仮説がある**: 「○○だと思う」と言えるだけの方向性がある
3. **答えを出せる**: 現実的に検証可能

「イシューの見極め」に時間を使う。イシューが筋悪なら、どれだけ調査しても成果が出ない。

### 5 Whys（トヨタ / 大野耐一）

「なぜ」を5回重ねて根本原因に至る。

例:

1. バグが出た → Q. なぜ？
2. テストが足りなかった → Q. なぜ？
3. テスト書く時間がなかった → Q. なぜ？
4. スプリントが詰まっていた → Q. なぜ？
5. 見積もりが楽観的 → Q. なぜ？
6. 過去データで見積もっていなかった

5回は目安。表層の症状で止まらないことが目的。

### MECE（Mutually Exclusive, Collectively Exhaustive）

漏れなくダブりなく分解。McKinsey / Barbara Minto の十八番。

- Mutually Exclusive: カテゴリが重複しない
- Collectively Exhaustive: カテゴリが全体を覆う

ロジックツリー / イシューツリーの分解ルール。

### フェルミ推定

ゼロから数値感覚を作る。調査前のサニティチェック、調査後の妥当性確認に使う。

例: 「このライブラリの採用企業数は？」→ GitHub Stars × 企業比率 × 実運用比率 で概算 → 実測と比較。

## 思考の方向付け

### 第一原理思考（First Principles）

アリストテレス → Elon Musk。既存の類推ではなく、還元できない事実から積み上げる。

例:

- 類推: 「バッテリーは高い」
- 第一原理: 「材料はコバルト・ニッケル・アルミ・炭素・プラスチック・鉄。市場価格はトンあたり X ドル。理論的最小コストは Y ドル」

ライブラリ評価で「このライブラリは重い」という評判を、依存ツリー・バンドルサイズ・実行時メモリ に分解して再検証するのが第一原理アプローチ。

### 反転思考（Inversion, Munger）

「成功する方法」より「確実に失敗する方法」を避ける方が容易。

- Pre-mortem: 「6ヶ月後この選択が失敗していたら、原因は何だったか」
- 「絶対にこのライブラリを選んではいけない場合は？」

### Steelmanning

相手の立場を**最強の形で**組み立ててから批判する（Strawman の対義）。

- ❌ Strawman: 相手の立場を弱く解釈して論破
- ✅ Steelman: 相手の立場を最善の形に再構築して、それでも反論できるか確かめる

確証バイアス対策の最強手段の一つ。

### Red Team Thinking

組織として反対仮説を検証する体制。軍事・セキュリティ由来。

- Blue Team: 自案を擁護
- Red Team: 自案を潰しにかかる

調査結果を上司に出す前に、自分で Red Team として叩くと、弱い箇所が先に見える。

### ベイズ更新

事前確率 × 尤度 → 事後確率。リサーチを「信念を段階的に更新する作業」と捉える。

新しい証拠が既存の信念をどれだけ動かすかを意識する。証拠1つで結論が大きく動くなら、それは脆い信念。

実践: 調査開始時に「今の確信度は X%」とメモし、新証拠ごとに更新する。

### Occam / Hanlon / Chesterton の剃刀

- **Occam's Razor**: 等しく説明できるなら単純な仮説を取る
- **Hanlon's Razor**: 愚かさで十分に説明できることに悪意を仮定するな（バグを陰謀と捉えるな）
- **Chesterton's Fence**: 「なぜあるか分からないフェンス」を撤去する前に、設置された理由を調べろ

特に Chesterton's Fence はレガシーコード調査で重要: 「なぜこの設定値があるか不明」「なぜこの回避策があるか不明」 → 撤去前に理由を遡る。

## システムと時間

### システム思考（Systems Thinking）

Donella Meadows『Thinking in Systems』。

- **ストック / フロー**: 貯蔵量と流入出量
- **フィードバックループ**: 正のループ（増幅）、負のループ（安定化）
- **レバレッジポイント**: 小さな介入で大きな変化を生む点（パラメータ < ルール < 目的 < パラダイム の順に強力）

単一変数の分析ではなく、相互作用を見る。

### 三現主義（現場・現物・現実）

トヨタ生産方式。抽象議論の前に現場へ行き、現物を見て、現実を把握する。

ソフトウェアでは:

- ログを見る（現実）
- production の実挙動を見る（現物）
- 実際のユーザーの画面を見る（現場）

「頭の中の想定」から離れる。

### 守破離

千利休 / 世阿弥。

1. **守** — 型を守る（ベストプラクティスをなぞる）
2. **破** — 型を破る（状況に応じて逸脱）
3. **離** — 型を離れる（独自の型を作る）

未知の領域はまず守で入る。いきなり離を目指すと車輪の再発明になる。

## Think/Plan/Act ループ

### OODA ループ（John Boyd）

- **O**bserve — 観察
- **O**rient — 状況判断
- **D**ecide — 意思決定
- **A**ct — 行動

OODA を速く回せる側が勝つ。リサーチでは「観察から判断までの速度」を意識する。

### PDCA（Deming）

- **P**lan — 計画
- **D**o — 実行
- **C**heck — 確認
- **A**ct — 改善

長期改善向き。

### Cynefin フレームワーク（David Snowden）

状況を5分類し、それぞれに適した意思決定方法:

- **Clear（明白）**: ベストプラクティス適用（Sense-Categorize-Respond）
- **Complicated（困難）**: 専門家分析（Sense-Analyze-Respond）
- **Complex（複雑）**: 実験して感知（Probe-Sense-Respond）
- **Chaotic（混沌）**: まず行動（Act-Sense-Respond）
- **Confusion（混乱）**: どこに分類すべきか不明 → まず分類する

ソフトウェア選定は Complicated（専門家分析）が多い。新領域のPoCは Complex（実験）。障害対応は Chaotic。

## 使い分けの目安

| 状況 | 使うフレームワーク |
|------|-------------------|
| 問いが曖昧 | イシュードリブン、5 Whys |
| 問いが複雑 | MECE、ロジックツリー |
| 既存類推に縛られている | 第一原理 |
| 確証バイアスが疑われる | Steelman、Red Team、Pre-mortem |
| 証拠が積み重なっていく | ベイズ更新 |
| 不確実性が高い | Cynefin、OODA |
| レガシーが気になる | Chesterton's Fence |
| 全体像が見えない | システム思考 |
