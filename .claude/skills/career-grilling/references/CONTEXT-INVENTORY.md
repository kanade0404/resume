# CONTEXT-INVENTORY: Phase 0 文脈接地の読む順序

Phase 0 は必須ゲート。ここを完了する前に Phase 1 以降へ進んではならない。
このファイルは「どのパスを・どの順で・どこまで読むか」だけを定義する。**実データ
(vaultの生の記述内容)はここには書かない** — このファイル自体が公開リポジトリに
コミットされるため。

## 0. vault パスの解決

1. `.claude/skills/career-grilling/vault-paths.local` を読む(gitignore対象)。
   フォーマットは `key=path` を1行1パスで、キーは `agent` / `personal` / `private`。
2. ファイルが無ければ、ユーザに一度だけ vault の実パスを尋ね、得られたら
   `vault-paths.local` に書き込む。以後のセッションはこのファイルを読むだけで
   再質問しない。
3. `private` パスがまだ `/add-dir` されていない場合、Phase 1 に進む前に一度だけ
   確認する:「add-dir しますか、それとも公開情報のみで進めますか」。自動 add はしない。

セクション0(このゲート)が完了するまで、セクション1(private vault の読み込み)を
開始してはならない。

以降、本ファイルでは `<private>` = private vault のルート、`<repo>` = 本リポジトリ
(`kanade0404/resume`)のルートを指す記号として使う。

## 1. 読む順序(具体パス)

| 順 | パス | 目的 | 読み方 |
|---|---|---|---|
| 1 | `<private>/profile/index.md` | 自己プロファイル全体のカタログ・エージェント注入ガイド | 全文 |
| 2 | `<private>/転職/<year>/グリルセッション成果_*.md` | 直近の壁打ち成果ログ | ファイル名を日付降順ソートし**最新1件のみ**全文 |
| 2' | `<private>/転職/<year>/grill-ledger.md` | セッション横断の既決着事項台帳 | 存在すれば全文([LEDGER-FORMAT.md](LEDGER-FORMAT.md) 準拠) |
| 3 | `<private>/profile/*.md` | 各トピックの現況 | **frontmatterのみ**(`sensitivity`/`status`/`updated`/`sources`)を一覧化。本文全文はPhase 3で該当ノードに降りたときに読む |
| 4 | `<repo>/docs/resume-update-plan.md` | 経歴書アップデート計画のギャップ一覧 | 全文 |
| 4' | `<repo>/docs/action-plan.md` | 期限付きアクションプラン | `- [ ]` の未完了タスクのみ抽出 |
| 5 | `<private>/daily report/**` (Tier4) | 内省差分サンプリング候補 | **全文走査しない**。ファイル名/更新日でサンプリング候補を列挙するだけに留め、Phase 4 で使う葉ノード候補としてリストアップする |

`year` は現在の活動年(セッション時点の年)を指す。過去年のディレクトリ
(`転職/2022-2023/`, `転職/2024-/` 等)は Phase 3 で矛盾突き合わせが必要になった
ときにのみ個別に開く。

## 2. profile stub 抽出

手順3で一覧化した frontmatter から `status: stub` または `status: needs-interview`
のファイルを抽出し、Phase 1 がルートを**内部選定**する際の候補材料として使う
(ユーザに選択肢として提示し選ばせるものではない — スコープ確認は事前承認ゲートとして
廃止済み)。

## 3. Tier4 サンプリング候補の扱い

Phase 0 ではファイルの存在と更新日だけを候補としてメモする(例:「特定の月に
記述の密度が高い」等の**メタ情報**まではよいが、本文の引用はしない)。Phase 4 で
実際に開いて profile 記述との差分を確認する。

## 4. 既知台帳(このセッション内)の作成

Phase 0 の成果物として、次の2種類をまとめた**既知台帳**を作る
([LEDGER-FORMAT.md](LEDGER-FORMAT.md) 準拠のスキーマを流用してよい):

- vault に既に書いてある事実(profile / 過去ログから拾えるもの)
- 過去ログ・grill-ledger.md で既に決着済みの決定(supersede されていないもの)

既知台帳は Phase 2 の決定木構築の入力になり、Phase 3 で「これは既知台帳で
決着済みか」を毎ノード確認することで、同じ問いの繰り返しを防ぐ。

ユーザに見せる既知台帳は**最大5行**に圧縮した要約とし(内部スクラッチはより詳しくてよい)、
末尾に「これらに変化があれば、この後の回答に添えるだけでよい」という注記を1行添える
([LEDGER-FORMAT.md](LEDGER-FORMAT.md) 参照)。これは返答を待つ確認質問ではなく注記であり、
独立の確認往復を作らない。既知事実の再掲でターンの大半を消費しない — 圧縮した要約は
Phase 1 の方針宣言・Phase 2 の Node 1 設問と**同じ1メッセージ**(セッションの初回
メッセージ)に収め、実質設問への到達を早める。
