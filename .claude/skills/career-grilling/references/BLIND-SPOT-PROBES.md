# BLIND-SPOT-PROBES: ジョハリの窓プロンプト集

盲点(自分では気づいていないが記録には残っている、または他者からは見えているが
本人が言語化していない領域)を探るための突合パターン集。Phase 0 で候補化し、
Phase 3/4 で実際に使う。

## 突合パターン

### 1. 日記 vs profile

Tier4(daily report等)の生の記述と、現在の `profile/*.md` の記述を突き合わせる。
「当時はこう思っていた/こう書いていたが、今の profile はそれと違う記述になっている」
という差分が見つかったら、偶発的な発見ではなく Phase 0 で候補化しておいたサンプルから
意図的に確認する。

質問の型: 「〈当時の記述の要旨〉と書いていた時期があるが、今の考え方はそれとは
違う。その転換点は何だったか」

### 2. README/detail.md vs 内的発話

経歴書本文(公開・整形済みの語り)と、壁打ち中にユーザが口にする本音や迷いを
突き合わせる。経歴書側の記述が「きれいにまとまりすぎている」箇所ほど、裏に
削ぎ落とされた迷いや失敗がある可能性が高い。

質問の型: 「`detail.md` ではこの実績は成功として書かれているが、実際に進めた
過程で迷った・失敗しかけた場面はあったか」

### 3. 他者評価 vs 自己評価

第三者からのフィードバック(リファレンスチェックで想定される評価、育成した
メンバーからの反応等、vault内に記録がある場合)と、本人の自己評価を突き合わせる。
本人が過小/過大評価している領域を特定する。

質問の型: 「もし当時一緒に働いていた人にこの点を聞いたら、あなたと同じ評価に
なると思うか。違うとしたらどう違うか」

### 4. 確度自己申告の内訳

自己申告された確度(%)の内訳を、楽観/悲観のどちらのバイアスが強く効いているかで
分解する。

質問の型: 「その確度の内訳のうち、自分でコントロールできる要素とできない要素は
それぞれ何か」

## 使い方の原則

- 突合の結果はDecision Journalの「決定/却下した代替案」に反映する
  ([DECISION-JOURNAL-FORMAT.md](DECISION-JOURNAL-FORMAT.md))
- 突合で得られた気づきは、必ずユーザ自身の言葉で言い直させてから記録する
  (フレームワークの分類名やこのファイルの言い回しをそのまま profile に転記しない)
