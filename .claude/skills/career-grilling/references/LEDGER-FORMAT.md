# LEDGER-FORMAT: 既知台帳(セッション内)と台帳ファイル(セッション横断)

過去セッションの最大の弱点は「セッション横断で同じ問いを繰り返す」ことだった
(2024年に作られた空のスタブファイルが2026年まで放置された例)。これを防ぐ2層の
台帳を定義する。

## 1. 既知台帳(セッション内、スクラッチ)

Phase 0 の成果物。ユーザに見せる1メッセージ、または scratch 領域のメモとして
持つ(gitignore対象、コミットしない)。スキーマ:

```
- topic: <一言でトピック名>
  status: fact | decided
  summary: <1行>
  source: <出典パス、行番号や見出し等>
```

`status: fact` は vault/コード/公開ドキュメントから読み取れる事実。
`status: decided` は過去の壁打ちでユーザが既に決定として確定させた事項
(grill-ledger.md や直近の成果ログから拾う)。

この台帳にある項目は Phase 3 で**再度聞かない**。木構築(Phase 2)の際、
既知台帳で埋まる葉ノードには決定を仮置きし、`resolved (from ledger)` として
扱う。

ただし「再度聞かない」は「その後の状況変化を確認しない」を意味しない。ユーザに
既知台帳を提示するメッセージの末尾には必ず「これらに変化があれば、この後の回答に
添えるだけでよい」という軽量な1行の**注記**を添える(全項目を1つずつ再質問する
のではなく、変化がある項目だけをユーザ自身に申告してもらう形)。この注記は返答を
待つ確認質問ではないため、これだけで独立したターンを作らない — 台帳提示とこの
注記は、Phase 2 で組み立てる実質設問(Node 1)と同じ1メッセージに同居させる。
確度%を伴う決定(例: 実現確度40%と自己申告された案件)は特に、時間経過でその後の
進捗が変わっている可能性が高いため、この注記を省略しない。

## 2. 台帳ファイル(`grill-ledger.md`、セッション横断・永続)

書き戻し先: `<private>/転職/<year>/grill-ledger.md`(private vault側、公開
リポジトリには置かない)。1トピック1行、日付降順で追記していく。

### スキーマ

| 列 | 内容 |
|---|---|
| topic | トピック名(profile のどのファイル/どの決定木の枝に対応するか) |
| 結論一行 | 決定の要約1行 |
| 日付 | `YYYY-MM-DD` |
| 出典リンク | 生ログファイルへの `[[出典]]` リンク(逐語はそちらにのみ存在) |
| supersedes | この行が上書きする過去の行(あれば)、無ければ空欄 |

### 例(フォーマットのみ、値はダミー)

```markdown
| topic | 結論一行 | 日付 | 出典リンク | supersedes |
|---|---|---|---|---|
| topic-A | 決定内容のダミー文字列 | YYYY-MM-DD | [[出典ファイルのダミーパス]] | - |
| topic-B | 決定内容のダミー文字列2 | YYYY-MM-DD | [[出典ファイルのダミーパス]] | - |
```

### 運用ルール

- **追記のみ、上書き削除しない**: 決定が変わったら新しい行を追加し `supersedes`
  列に旧行のトピック名+日付を書く。矛盾する2行が残ることを許容し、履歴として読める
  状態を保つ
- **出典リンクは必須**: 逐語・導出過程を書かず、生ログへのリンクのみ持つ
  (詳細は [WRITEBACK-SCHEMA.md](WRITEBACK-SCHEMA.md) の二層構造)
- 次回セッションの Phase 0 はこのファイルを読み、`supersedes` されていない最新行だけを
  既知台帳に取り込む
