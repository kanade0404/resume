# Data Model: Immutable Data Model (kawasima) + TM法 (佐藤正美)

データの「性格」を分類してモデルを切り分けるレンズ。RDB / アプリ層共通で効く。

## Immutable Data Model（kawasima）

属性を性格で分けると、自然と不変な核と可変な周辺に分離される。

| 性格 | 例 | 扱い |
|---|---|---|
| **不変属性** | identifier, 生年月日, 注文時刻 | 設定後は更新禁止。書き込み 1 回。 |
| **可変属性** | 状態, 残高, 名前変更可能ユーザ名 | 履歴と一緒に保つ（後述） |
| **履歴属性** | 過去の名前, 過去の住所, 過去の状態 | 別テーブル / 別集約に分離 |
| **派生属性** | 集計値, 計算済キャッシュ | 永続化するなら投影として明示。 |

### ルール

- **不変属性と可変属性を 1 テーブル / 1 クラスに混ぜない。** 混ざるとロックの単位、監査の単位、キャッシュの単位がすべて壊れる。
- **可変属性は履歴を残せる構造にする。** "現在の値" だけを上書きするのは情報損失。`<entity>_history` テーブル / event ログ / valid_from-valid_to を初期から検討する。
- **削除は論理削除でも物理削除でもなく "状態遷移として表現"。** "削除済"を `deleted_at` でなく `Status = Deleted(at)` で表す。
- **更新時刻列に意味を持たせない。** `updated_at` は監査用 metadata。これでドメインロジックを駆動しない。
- **"履歴を持つ" と "Event Sourcing" は別物。** 履歴属性 = "現在 + 過去の値"、Event Sourcing = "全変更の事象列から状態を再生"。レンズが違う（後者は `references/architecture.md`）。

## TM法（佐藤正美）

「事実とは何か」を主キーで強制するモデリング。要点だけ：

### Entity の 2 種類

- **Resource**: そこに在るもの。`User`, `Product`, `Account`。
- **Event**: 起こったこと。`Order`, `Payment`, `Login`。

両者は性格が違う。混ぜると主キー設計と整合制約がぶれる。

### 主キー規律

- **後から変わる属性を主キーに使わない。** email や名前を natural key に据えない。
- **事実が確定する瞬間に主キーが決まる。** Event は「起こった瞬間」、Resource は「登録された瞬間」。
- **複合主キーを避けない。** Event の主キー = (Resource1, Resource2, 時点) になることがあり、それを surrogate id 1 列に潰すと整合性チェックが効かなくなる。
- **null は事実の欠損であって "値" ではない。** null を意味のある状態にしない（"未設定" を nullable で表すのではなく、状態遷移で表す）。

### 認識と事実の分離

- **同じ実体を別の文脈が別の名前で呼ぶことがある**（DDD の Bounded Context と整合）。事実モデルは context-free の核を保ち、context ごとの projection を作る。
- **2 つの事象が同時に起きうるか?** 起きうるなら主キー候補に時刻だけでは足りない（順序保証が必要）。

## 値オブジェクトとプリミティブ強制

`references/security.md` の "ドメインプリミティブ" と接続する。Email, Money, OrderId, ZipCode などはプリミティブ型のままにせず値オブジェクトに昇格する。基準：

- 同じ型の 2 値を取り違える危険がある（`int customerId, int productId` を入れ替えてもコンパイルが通る）
- 妥当性ルールがある（Email の正規表現、Money の通貨整合）
- ドメイン固有の演算がある（`Money + Money` だが `Money + int` は不可）

## レビュー観点

- 「不変・可変・履歴」が 1 つのモデルに混ざっていないか
- 主キーが「後から変わる属性」を含んでいないか
- nullable 列が "状態" を表していないか（→ ADT に直す）
- Event Resource を 1 テーブルにまとめていないか
- DateTime 列の意味が「監査」「事実発生時刻」「ビジネス締切」のどれか命名で分かるか
