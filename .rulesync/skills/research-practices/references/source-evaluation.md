# 情報源の評価

情報源の信頼性を見極める判断材料。SKILL.md §3 の信頼度タグを付ける前に、各ソースをここの基準で評価する。

## エビデンス階層（ソフトウェア領域向け）

医療のEBMエビデンス階層をソフトウェアに翻案:

| レベル | 医療版 | ソフトウェア版 | 信頼度タグ |
|-------|-------|---------------|-----------|
| 最高 | Systematic Review / Meta-analysis | 複数事例を統合した大規模サーベイ（State of JS/CSS、Stack Overflow Survey、ACM 文献レビュー） | S1 |
| 高 | RCT | 統制されたベンチマーク、A/Bテスト、再現可能な実験 | S0-S1 |
| 中 | コホート研究 | 長期運用レポート、本番事例研究、大規模ユーザーの事例報告 | S2 |
| 中 | ケーススタディ | 単一事例ブログ、カンファレンストーク | S2-S3 |
| 低 | 専門家意見 | ツイート、Stack Overflow 回答、Discord/Slack の発言 | S3 |
| 最低 | 逸話 | Reddit 単発コメント、無署名の短文 | S4 |

LLM 出力（S5）は独立枠として扱う。検証すれば上のレベルに昇格、未検証なら S5 のまま。

## 一次 / 二次 / 三次ソース

- **一次（Primary）**: 公式仕様、RFC、論文本体、ソースコード、CHANGELOG、公式ドキュメント
- **二次（Secondary）**: 解説記事、レビュー論文、教科書、メンテナ以外による技術ブログ
- **三次（Tertiary）**: Wikipedia、Cheatsheet、まとめ記事、AI 生成のサマリ

**重要な主張は一次まで遡る**。二次・三次は「案内図」として使う。一次情報を読んでいないまま結論を書かない。

## CRAAP テスト

各ソースについて:

- **C**urrency — いつ書かれたか、対象バージョンは何か、最新か
- **R**elevance — 自分の問いに対して関連性があるか（関連しない詳細は捨てる）
- **A**uthority — 著者は誰か、何の権威・経験で書いているか
- **A**ccuracy — 主張は検証可能か、一次情報への引用があるか
- **P**urpose — なぜ書かれたか（広告 / 教育 / 主張）、利益相反はあるか

CRAAP のどれかが欠けているなら信頼度を下げる。全部埋まって初めて高信頼。

## SIFT（Mike Caulfield）

プロのファクトチェッカーが使う4ステップ:

1. **S**top — 反応する前に止まる（感情的に強く反応するコンテンツほど操作されている可能性）
2. **I**nvestigate the source — 発信者を先に調べる（所属、過去の発言、背景）
3. **F**ind better coverage — 同じ主張が他ソースで裏付けられるか横断確認
4. **T**race claims — 引用・画像・データを元に遡る（引用元が主張を支えていないケース多数）

## Lateral Reading

1ページを上から下に読むのではなく、**複数タブを開いて横断的に評価する**。

- 発信サイト自体の信頼性を別のソースで確認
- 主張を別ソースでも確認
- 逆の主張をしているソースがあるか確認

これが「垂直に読む」よりプロフェッショナルな方法として確立されている。

## ピアレビューとプレプリント

- **査読済み論文**: 分野ごとに査読精度は違うが、一定の品質フィルタを通過している
- **プレプリント（arXiv, bioRxiv, SSRN）**: 速報性は高いが未査読。引用時は明示する
- **Retraction Watch**: 撤回済み論文の追跡。重要な引用前にチェックする価値あり
- **プレプリントから査読版への変更点**: 最終版と異なる主張をしていないか確認

## ベンチマーク数値の読み方

ベンチマークは特にバイアスに汚染されやすい。以下を確認:

- **誰が測ったか**: ベンダー自社測定は割り引く（COI）
- **何を測ったか**: マイクロベンチはしばしば現実の負荷と乖離
- **どう測ったか**: ウォームアップ、GC、JIT、キャッシュ効果、外れ値処理
- **再現可能か**: コード・データセット・環境・ハードウェアが公開されているか
- **自分のユースケースに近いか**: 全く違うワークロードなら参考程度

**Goodhart's Law**: 「指標が目標になると、それは良い指標でなくなる」。ベンチが最適化対象になっている可能性を常に疑う（例: TechEmpower Round 24 向けにチューニングされたフレームワーク）。

## 灰色文献（Grey literature）

白書、テクニカルレポート、企業エンジニアリングブログ、内部報告、カンファレンス資料。査読はされないが、ソフトウェア分野では重要な一次情報。

特に有用:

- Netflix Tech Blog, Stripe Engineering, Shopify Engineering, Uber Engineering, Discord Engineering
- Google / Meta / Amazon のカンファレンス資料
- 公開されたポストモーテム

## 日本語情報の扱い

Streetlight Effect 対策で、英語圏だけに偏らないこと。特に:

- **CiNii, J-STAGE**: 日本語学術論文
- **Qiita, Zenn**: 日本語エンジニアコンテンツ
- **JapanTaxi / メルカリ / サイバーエージェント** 等の国内エンジニアリングブログ

日本語一次情報がある場合、それを無視して海外二次情報だけに依存するのは偏りの温床。
