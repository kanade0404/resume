# Test-Driven Development (Kent Beck)

TDD は「テスト戦略」ではなく **設計圧をかける手順**。設計レンズとして使う。

## 基本リズム

1. **Red**: 失敗する小さなテストを 1 本書く（まだ実装はない）。
2. **Green**: テストが通る最小の実装を書く。汚くて良い。
3. **Refactor**: 通ったまま設計を改善する（重複除去、命名、構造）。

green を直接書かない。red を経由しないテストは「実装に追従させた仕様」になり、保護にならない。

## 設計圧として何が起きるか

- **テストが書きにくい = 設計が間違っている**。Test Double が欲しい時点で Functional Core 抽出できていない（`functional-core.md`）。
- **公開 API しか叩けない**。private を直接テストしたくなったら、その private は公開される価値がある別概念。
- **依存が引数で渡される**。global / singleton にすると test fixture の組み立てが地獄になる。

## 適用範囲

- **Core ドメインに必須**。Supporting は薄ければ後追いテストでも可。Generic はそもそも自前でテストしない。
- **インフラに直結する I/O は Humble Object 化**してから純関数を TDD する（Functional Core 部分）。
- **UI / レイアウトは TDD の対象として向かない**（視覚的判断は人間レビュー / ビジュアルリグレッション）。

## TDD と Refactor の作法

- **3 strikes and refactor.** 同じ知識が 3 回出てきたら抽象化候補。1 回 / 2 回では抽象化しない（過剰一般化）。
- **Refactor は green を保ったまま小刻みに。** 一度に複数変更しない。
- **設計の "形が見えた瞬間" にだけ抽象化する。** 想像で前もって作らない（YAGNI）。

## Beck の "Tidy First?" 視点

- リファクタリングを **構造変更（tidy）** と **振る舞い変更** に分ける。1 コミットで両方やらない。
- レビューが速くなる。差分の意図が読める。

## TDD で書きたくなくなる兆候 = 設計のアラート

| 症状 | 設計の疑い | 直し方 |
|---|---|---|
| Test Double を 5 個以上組まないとテストが書けない | Functional Core / Imperative Shell が分離できていない | 純関数を抽出 |
| 1 つのテストが 30 行以上ある | 集約が大きすぎる / 判断と I/O が混在 | 集約分割 / Humble Object 化 |
| `private` を露出してテストしたい | private が別概念。集約の責務が混ざっている | 切り出し |
| time / random / uuid をテストで触れない | DI されていない | Clock / IdGenerator Protocol を引数化 |
| テストが flaky | 副作用 / 順序依存 / 共有可変状態 | `test-review` のフレーキー分類へ |

## レビュー観点

- 新規 Core 機能に対して「どのテストから書いたか」が辿れるか（コミット粒度）
- green を経由せずいきなり仕上がっているテストになっていないか（red のコミットが見える）
- Test Double の濫用がないか（Test Double が要る時点で設計を疑う）
- リファクタリングコミットと振る舞い変更コミットが分離されているか
