# Secure by Design (Bergh Johnsson, Deogun, Sawano)

「セキュリティはドメインモデルの中で表現する」というレンズ。後付けレイヤや middleware ではなく、**型と値オブジェクトに契約を持たせる**。

## 中核命題

- **Security and Domain Modeling are the same activity.** モデル設計が緩いとセキュリティも緩む。逆にモデルが厳格なら、後付けの境界バリデーションを大量に書かなくて済む。
- **Validation should not live at "the boundary".** "API 入口で全部チェックする" 戦略は層が増えると破綻する。**入口を通れる時点で値が妥当である** のが理想。
- **失敗時は閉じる方向に倒す（fail securely / fail closed）。** 例外時 / 想定外時に "とりあえず通す" / "とりあえず権限を緩める" を default にしない。

## ドメインプリミティブ（中核パターン）

- **値オブジェクト + コンストラクタで全契約を強制。**
- 例: `Email`, `Money(amount, currency)`, `OrderId`, `ZipCode`, `Username`, `PasswordHash`。
- ルール:
  - **不変**（コンストラクタで作って以降変更不可）
  - **コンストラクタが妥当性を検証**（不正値は構築不能 → 例外 or `Result`）
  - **同じプリミティブ型 2 値の取り違え不能**（`int customerId` と `int productId` を混同しない）
  - **危険な操作はインスタンスメソッドだけが提供**（生 `string` を露出しない）
- バリデーションが "層を貫いて散る" のを防ぐ。1 度通った値は型レベルで保証される。

## 制約値クラス（範囲付きプリミティブ）

- 定義域が決まっている整数や数値はラップする（`Quantity`（1〜9999）、`Percentage`（0〜100））。
- DB / フォーム / API のあらゆる経路で同じ範囲チェックを書かない。

## 認可をドメイン操作の前提に組み込む

- 「権限がないと操作が呼べない」を **型 / 引数で表す**。
  - 操作が `Capability` 引数を要求し、Capability は authorization layer でしか取れない、など。
  - 「メソッド内で role を見て if 分岐」は authorization smell。
- ドメインメソッドは "渡された capability から必要な権限を持つことが既知" として実行する。

## 機微データの暴露を型で塞ぐ

- `SensitiveString`, `Secret<T>` のような型を作り、`toString` / ログ出力 / シリアライズで自動的に redact。
- 平文をログに出さないようにレビュー観点で機械チェックできる（「`SensitiveString` を `String` 経由で touch していない」を grep）。

## 入力境界での明示的変換

- 外部 (HTTP, queue, file) から来た raw 値は **必ずドメインプリミティブに変換してから内部に渡す。**
- 変換失敗は `Result<T, ValidationError>` か例外で。**raw 値が内部に流れ込まないように型で阻止。**

## "悪質な入力" を扱うパターン

- **Quantity に上限を必ず置く**（`Quantity.MAX = 1_000_000`）。「無制限」を許容するとリソース枯渇攻撃が通る。
- **Collection size の上限**（リクエストのアイテム数、ファイルサイズ、ペイロード長）。
- **時間 / レート / 試行回数の上限**（指数バックオフ + 上限）。
- **Idempotency key の必須化**（重複処理によるサイドエフェクト爆発を防ぐ）。

## Fail securely

- 認可判定: 例外時 → deny。
- ロック / 排他: 例外時 → 取得失敗扱い（"取れていないのに進む" にしない）。
- フィーチャートグル: 不明時 → off（特に "破壊的" な機能）。

## 監査と非否認

- ドメイン操作のログは **誰が（identity）・いつ・何を（domain operation）・結果（Result の枝）** を必ず含める。
- ログは可逆で読み手のあるドキュメント。`trace_id` / `user_id` を必ず付ける。
- 機微データはログに出さない（前述の `Sensitive` 型）。

## レビュー観点

- 入力 raw 値が API 入口より深く流れていないか
- 値オブジェクトを作るタイミング基準（型混同 / 妥当性 / 演算）が満たされる場所でプリミティブのままになっていないか
- バリデーションが境界 / middleware / service / DB に重複していないか
- 認可判定がドメイン操作の中の if で書かれていないか
- 失敗時のデフォルト動作が "通る" になっていないか
- Quantity / collection size / 試行回数の上限が定義されているか
- ログに平文の機微データが乗っていないか
