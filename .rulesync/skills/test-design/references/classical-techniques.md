# 古典的テストケース設計技法

同値分割・境界値分析・デシジョンテーブル・状態遷移テスト・ペアワイズ/組合せテストの手順と選択基準。出典は Glenford Myers『The Art of Software Testing』および ISTQB シラバスの標準的な定義に準拠する。

各節の末尾に「このケースをどう設計表に落とすか」の対応を書く。

---

## 1. 同値分割 (Equivalence Partitioning)

**定義.** 入力空間を「同じ振る舞いをするはず」のクラスに分割し、各クラスから代表値を 1 つ選んでテストする。全入力を試す代わりに、クラスの数だけ試せば十分という仮定に基づく。

**手順.**
1. 入力パラメータごとに、仕様が振る舞いを変える境目を洗い出す。
2. 各境目で **有効クラス (valid partition)** と **無効クラス (invalid partition)** に分ける。無効クラスは 1 つのパラメータにつき 1 つずつ独立にテストする（複数の無効値を同時に混ぜると、どちらが原因で失敗したか切り分けられなくなる）。
3. 各クラスから代表値を 1 つ選ぶ（境界値自体は BVA に譲り、クラス内の「中央値」を選ぶ）。

**例.** 年齢を受け取り「未成年/成人/シニア」を返す関数:
- 有効クラス: `[0, 17]` → 未成年、`[18, 64]` → 成人、`[65, 150]` → シニア
- 無効クラス: 負数、151 以上、非数値

**設計表への落とし方.** 技法列 = `同値分割`、入力列 = 選んだ代表値、導出根拠列 = `同値クラス: <クラス名>`。

## 2. 境界値分析 (Boundary Value Analysis, BVA)

**定義.** 欠陥は境界の近くに集中するという経験則に基づき、各境界の **on-point**（境界値そのもの）と **off-point**（境界のすぐ外側）を両方テストする。

**手順.**
1. Step 1 で明文化した範囲・長さ・件数の境界を全て列挙する（最小値・最大値・0・空・上限件数など）。
2. 各境界について次の値を候補にする:
   - **on-point**: 境界値そのもの（`min`, `max`）
   - **off-point**: 境界から、入力ドメインで意味を持つ最小の刻み幅だけ外側の値。整数・文字列長・配列長など離散カウントのドメインなら `±1`（`min - 1`, `min + 1`, `max - 1`, `max + 1`）。小数・金額・日付時刻など連続量のドメインでは `±1` を機械的に使わず、そのドメインの最小意味単位（例: 金額なら最小通貨単位=1円/1セント、日時なら扱う精度の最小単位、浮動小数点なら丸め誤差を超える epsilon）を刻み幅にする。
   - **特殊値**: `0`、空文字列/空配列、`null`/`None`、単一要素
3. 単純 BVA では代表的な境界のみ、**worst-case BVA** では複数パラメータの境界を掛け合わせる（パラメータ数が少なく組合せ数が現実的な場合のみ推奨、多い場合は §5 のペアワイズに切り替える）。

**例.** 文字列長 1〜100 を受け付けるバリデータ:
- on-point: 長さ 1, 長さ 100
- off-point: 長さ 0（無効）, 長さ 2, 長さ 99, 長さ 101（無効）
- 特殊値: 空文字列（長さ 0 と重複するので 1 ケースに統合してよい）

**設計表への落とし方.** 技法列 = `BVA`、導出根拠列 = `境界: <パラメータ名>の<min/max> <on/off>`。

## 3. デシジョンテーブル (Decision Table Testing)

**定義.** 複数条件の組合せで分岐が決まる振る舞いを、条件×アクションの表で網羅し、冗長な列を簡約する。

**手順.**
1. 分岐に関わる **条件 (condition)** を全て列挙する（真偽値または少数の離散値に正規化する）。
2. 条件数を `n` として `2^n` 通り（離散値ならその積）の組合せ列を作る（フルテーブル）。
3. 結果に影響しない条件（**don't care**）をまとめて列を簡約する。簡約後の列数が実際に書くべきテストケース数になる。
4. 各列に対応するアクション（期待される出力・副作用）を書く。

**例.** 割引率の決定（会員種別: 一般/プレミアム、購入額: 1万円未満/以上）:

| 会員種別 | 購入額 | 割引率 |
|---|---|---|
| 一般 | 1万円未満 | 0% |
| 一般 | 1万円以上 | 5% |
| プレミアム | 1万円未満 | 10% |
| プレミアム | 1万円以上 | 20% |

条件が増えるほど `2^n` が爆発するため、無関係な条件の組が判明した時点で簡約する。**ペアワイズへの切り替えは、デシジョンテーブルが要求する全アクション（3 条件以上の相互作用で初めて発生する特殊なアクションを含む）を最低 1 回ずつ網羅した後にのみ行う** — ペアワイズは任意の 2 因子間の組合せ網羅を保証するだけで、3 因子以上の相互作用でしか現れないルール列の網羅は保証しない。条件数が多い場合でも、まず簡約済みデシジョンテーブルの各列（アクション）を最低 1 ケースずつ確保し、それ以外の独立した入力次元の間引きにのみ §5 のペアワイズを使う。

**設計表への落とし方.** 技法列 = `デシジョンテーブル`、導出根拠列 = `条件列: <条件の組合せ>`。

## 4. 状態遷移テスト (State Transition Testing)

**定義.** 対象がライフサイクル・状態機械を持つ場合、状態遷移図から遷移網羅の基準に従ってケースを導出する。

**手順.**
1. 状態と、状態を変えるイベント（トリガ）を洗い出し、状態遷移図または状態遷移表を書く。
2. カバレッジ基準を選ぶ:
   - **0-switch カバレッジ**: 全ての個別遷移（1 本のエッジ）を最低 1 回通す。
   - **1-switch カバレッジ**: 連続する 2 遷移の全ペアを通す（0-switch より厳密、遷移の順序依存バグを検出しやすい）。
3. 各遷移について、正当な遷移だけでなく **不正な遷移**（その状態で許可されていないイベント）もケース化する — 何も起きない/エラーになることを確認する。

**0-switch/1-switch の既定基準.** 対象がステートレス（状態を引数で受け取り新状態を返すだけの純関数）なら 1-switch は 0-switch の合成に還元されるため省略可。対象がメソッド呼び出しで隠れた内部状態を書き換えるオブジェクトなら 1-switch（連続 2 遷移）を必須とする。

**例.** 注文の状態機械（`pending → paid → shipped → delivered`、`pending/paid → cancelled`）:
- 0-switch: `pending→paid`, `paid→shipped`, `shipped→delivered`, `pending→cancelled`, `paid→cancelled` の 5 遷移それぞれ
- 1-switch: `pending→paid→shipped` のような連続 2 遷移のペア
- 不正遷移: `shipped` 状態で `cancelled` イベントを送っても状態が変わらない/拒否されることを確認

**設計表への落とし方.** 技法列 = `状態遷移`、導出根拠列 = `遷移: <from>→<to> (0-switch/1-switch)`。

## 5. ペアワイズ / 組合せテスト (Pairwise / Combinatorial Testing)

**定義.** パラメータ数が多く全組合せ (`n1 × n2 × ... × nk`) が非現実的な場合、実務上の欠陥の大半は 2 パラメータ間の相互作用で発生するという経験則に基づき、全ペアが最低 1 回は出現する縮小組合せでカバーする。

**手順.**
1. 各パラメータの水準（値の代表クラス、同値分割/BVA で既に絞ったもの）を列挙する。**ここに含めるのは有効クラス (valid partition) の代表値のみ**。無効クラスは §1 の原則（1 パラメータにつき 1 つずつ独立にテストし、複数の無効値を同時に混ぜない）に従い、ペアワイズの組合せ集合には含めず別枠のケースとして追加する。
2. 全パラメータの「任意の 2 つの組合せ」が最低 1 回出現する最小の組合せ集合を作る（直交表やペアワイズ生成アルゴリズムに委ねる。具体的なツール実行は本スキルの範囲外 — 生成ロジック自体は実行者やツールに任せる）。
3. **全組合せが必要な場合との使い分け**: 次のいずれかに該当するなら間引かず全組合せにする。
   - 安全性・金銭・認可に関わり、組合せ漏れの発見コストが極めて高い
   - パラメータの相互作用が 3 パラメータ以上に及ぶ既知の実績がある（過去に 3-way 以上の相互作用バグが出た領域）
   - パラメータ数が少なく（3 個以下など）全組合せがそもそも小規模

**設計表への落とし方.** 技法列 = `ペアワイズ` または `全組合せ`、導出根拠列 = `組合せ: <どのパラメータの組か>`。どちらを選んだかと、その理由（リスク vs. 本数）を必ず書く。
