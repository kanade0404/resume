# Test Sizes 配分（Google Software Engineering）

出典: *Software Engineering at Google*（Winters, Manshreck, Wright 編, O'Reilly, 2020）第 11 章。テストピラミッド（unit/integration/E2E のような**アーキテクチャ層**による分類）とは軸が異なり、Test Size は **実行環境の制約**（プロセス境界・ネットワーク・時間）による分類。本 reference は「書く前にどのサイズに割り当てるか」の判断に限定し、書かれたテストの事後レビューは `test-review` §9 に譲る。

---

## 定義

| Size | プロセス境界 | ネットワーク | 実行時間の目安 | 典型的な特徴 |
|---|---|---|---|---|
| **Small** | 単一プロセス内 | 禁止（localhost も不可） | 数十 ms〜1 秒 | 決定的、hermetic、外部 I/O ゼロ |
| **Medium** | 単一マシン内で複数プロセス可 | localhost / ループバックのみ許可 | 数秒 | ローカル DB・ファイルシステム・ローカルプロセス間通信を許容 |
| **Large** | 複数マシン可 | 実ネットワーク許可 | 数秒〜数分 | 外部サービス・実クロック・実タイムアウトを含みうる |

分類の基準は「テストが何に依存して落ちうるか」であって、テスト対象のコード量ではない。1 行の関数でもネットワーク越しに実行すれば Large になり、100 行のロジックでも純粋な入出力変換なら Small になる。

---

## どの振る舞いをどのサイズに置くか

Step 4 でケースごとに判断する基準:

1. **外部 I/O があるか** — DB・ファイル・ネットワーク・外部プロセスに触れるなら Small から外れる。
2. **時間・並行性に依存するか** — 実クロックや実タイムアウトに依存するなら Medium/Large。決定的な Clock を注入できるなら Small のままにできる（`test-review` §6 の Clock DI 参照）。待機・ポーリングを伴わない一回性の時刻比較（有効期限判定など）であっても、実際のシステム時刻をそのまま読む設計は非決定的であることに変わりないため、Small に置くなら必ず注入可能な Clock を使う——閾値に余裕を持たせるだけでは hermetic 性を代替できない。
3. **失敗時の原因特定コスト** — Large は失敗要因が多く原因特定が高コスト。同じ振る舞いが Small で検証できるなら、Small のケースを優先し Large は「Small では検証できない統合面」だけに絞る。

**設計表への反映.** Step 2〜3 で技法から導出したケースのうち、契約検証・property・BVA・同値分割・デシジョンテーブルの大半は純粋な入出力変換として Small に落とせる。状態遷移テストで実際の永続化を伴う場合は Medium。複数サービスをまたぐ E2E 相当のシナリオのみ Large にする。

## Change-detector test を書かない

**定義.** 実装の内部手順・呼び出し順・中間状態をそのまま「写経」したテスト。仕様が変わっていなくても、実装のリファクタだけで壊れる。

**書く前に避ける方法.**
- 設計表の「期待」列には、常に **観測可能な出力・状態** だけを書く（戻り値、永続化された行、外部に送られたメッセージの内容）。呼び出し回数・呼び出し順・内部関数の呼び出し有無を期待列に書かない。
- 契約の `Q`（事後条件）を直接転記する。実装コードを読んでから「どう動いているか」を写すのではなく、Step 1 で先に契約を書いてからケースを導出する順序を守る（実装を読まずに契約が書けるなら change-detector になりにくい）。
- property-based のケースは性質そのものが期待値になるため、原理的に change-detector になりにくい。境界値・同値分割のケースで「実装の分岐に合わせて期待値を作る」誘惑が出たら、契約の `Q` に立ち返る。

## Flaky になりにくい設計（Hermetic）

書く前の設計段階で次を満たすケースだけを Small/Medium に割り当てる。満たせないなら Large に置くか、そのケース自体の設計（対象の I/O 境界）を疑う。

- **hermetic**（外部環境から隔離されている）: 他のテストの実行順序・他プロセスの状態に依存しない。
- **決定的な入力生成**: 乱数・現在時刻を使う場合は seed / 注入可能な Clock を前提にケースを設計する（具体的な DI 手法は `test-review` §6 の flakiness 原因分類を参照）。
- **タイムアウト依存を避ける**: 「N 秒待って完了しているはず」という設計のケースは避け、完了イベントを observable にしてから assert する設計に倒す。

Small/Medium/Large の内訳は設計表末尾の「Test Size 内訳」に集計し、Large が多すぎる（目安: 全体の 1 割を超える）場合は、Step 2 に戻って対象の I/O 境界そのものを見直す余地がないか確認する。
